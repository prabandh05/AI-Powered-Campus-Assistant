1. PROJECT OVERVIEW
================================================================================

Campus AI OS is a Role-Based AI-Powered Campus Assistant built for Dayananda
Sagar College of Engineering (DSCE). It combines a traditional SQL database for
structured data (student profiles, faculty schedules) with a RAG (Retrieval-
Augmented Generation) engine for unstructured campus knowledge, delivering an
intelligent assistant that can answer any question about the college.

The system is designed with three user roles — Student, Faculty, and Admin —
each with tailored dashboards and permissions. The frontend is built with React
(Vite) and styled to look like a modern AI chat application (inspired by ChatGPT
and Claude AI interfaces).


2. PROBLEM STATEMENT
================================================================================

Students and faculty at DSCE need quick, reliable answers about:
  - Admission procedures, fee structures, eligibility criteria
  - Department information, course details, syllabus
  - Placement statistics, recruiter lists
  - Faculty availability and consultation hours
  - Events, notices, rules and regulations
  - Campus facilities (hostel, library, sports, etc.)

Traditional methods (browsing the website, asking seniors, visiting offices)
are slow and often produce incomplete answers. This system centralizes all
campus knowledge into a single AI-powered interface.

4. TWO TYPES OF QUERIES (WHY HYBRID)
================================================================================

  TYPE 1: STRUCTURED / EXACT DATA (→ SQL Database)
  ─────────────────────────────────────────────────
  Examples:
    - "What are my marks?"           → Fetches from StudentProfile table
    - "Is Shreya available?"         → Fetches from StaffProfile table
    - "Show me faculty schedule"     → Fetches from StaffSchedule table

  These need PRECISION, not AI guessing. An LLM cannot accurately track
  schedules or grades. SQL handles this perfectly.

  TYPE 2: UNSTRUCTURED / KNOWLEDGE (→ RAG Pipeline)
  ─────────────────────────────────────────────────
  Examples:
    - "What are attendance rules?"   → Searches campus knowledge base
    - "Tell me about placements"     → Retrieves placement statistics
    - "What is the admission process?" → Returns eligibility criteria

  These need SEMANTIC UNDERSTANDING. The user's question may not exactly
  match the stored text, so we use vector embeddings to find the most
  similar content regardless of exact wording.

  WHY NOT JUST USE AN LLM?
  - LLMs hallucinate (make up facts)
  - LLMs can't track real-time faculty availability
  - LLMs can't query a database for exact marks
  - LLMs don't know college-specific data unless you feed it

  SOLUTION: We SPLIT the responsibilities. The Query Router classifies
  every user question and sends it to the right engine.


5. RAG (RETRIEVAL-AUGMENTED GENERATION) PIPELINE
================================================================================

  Step 1: DATA COLLECTION
  ───────────────────────
  - A web scraper (BeautifulSoup) crawls the DSCE website (dsce.edu.in)
  - Extracts text from 50+ pages (departments, admissions, placements, etc.)
  - Raw data saved to: data/campus_knowledge.txt (1.3 MB, 50,189 lines)

  Step 2: DATA CLEANING
  ─────────────────────
  - Script: scraper/clean_data.py
  - Removes navigation menus, footers, boilerplate (90% of raw data is noise!)
  - Deduplicates repeated lines
  - Output: data/campus_clean.txt (347 KB, 4,907 clean lines)

  Step 3: CHUNKING
  ────────────────
  - Paragraph-aware chunking (not naive word splitting)
  - Groups related paragraphs together (300-400 words per chunk)
  - Preserves semantic coherence within each chunk
  - Avoids splitting mid-sentence

  Step 4: EMBEDDING
  ─────────────────
  - Model: sentence-transformers/all-MiniLM-L6-v2
  - Converts each text chunk into a 384-dimensional vector
  - Vectors normalized for cosine similarity

  Step 5: INDEXING
  ────────────────
  - FAISS (Facebook AI Similarity Search) index
  - Inner Product search on normalized vectors = cosine similarity
  - Current index: 224 vectors (chunks)
  - Stored at: data/vector_db/faiss.index + data/vector_db/chunks.npy

  Step 6: RETRIEVAL + GENERATION
  ──────────────────────────────
  - User asks a question
  - Question is embedded using the same model
  - FAISS finds the top-5 most similar chunks
  - Chunks are passed as context to the LLM (Groq Llama-3.1-8b)
  - LLM generates a grounded answer using ONLY the provided context
  - Anti-hallucination rules in the system prompt prevent fabrication


6. QUERY ROUTING SYSTEM
================================================================================

  The QueryRouter classifies every user query using fast keyword matching:

  ┌─────────────────────┐
  │   User Query         │
  └──────────┬──────────┘
             │
  ┌──────────▼──────────┐
  │  Does it mention     │──YES──→ SQL (database lookup)
  │  "my marks", "cgpa"? │
  └──────────┬──────────┘
             │ NO
  ┌──────────▼──────────┐
  │  Does it mention     │──YES──→ HYBRID (SQL + RAG)
  │  "meet", "faculty",  │
  │  "available"?        │
  └──────────┬──────────┘
             │ NO
  ┌──────────▼──────────┐
  │  General knowledge   │──YES──→ RAG (vector search)
  │  question?           │
  └──────────┬──────────┘
             │ NO
             └──────────────────→ RAG (default fallback)

  SQL keywords: my marks, my profile, my cgpa, is available, can i meet, etc.
  HYBRID triggers: faculty, meet, staff, available, mam, sir, professor
  RAG keywords: placement, admission, hostel, department, exam, events, etc.


7. ROLE-BASED ACCESS CONTROL
================================================================================

  STUDENT ROLE:
    - Can chat with the AI assistant (RAG + SQL queries)
    - Can view their own profile (marks, CGPA, department)
    - Cannot access admin or faculty panels

  FACULTY ROLE:
    - Can chat with the AI assistant
    - Can edit their profile (name, department, designation)
    - Can toggle availability status (available / not available)
    - Can set consultation hours (when students can meet them)

  ADMIN ROLE:
    - Can chat with the AI assistant
    - Snap Knowledge: Inject quick facts directly into the vector database
    - Formal Knowledge: Add structured data with title, category, and content
    - File Upload: Upload PDF/TXT files to expand the knowledge base
    - System Dashboard: View all registered users and indexed documents

  Authentication: JWT tokens (30-minute expiry), password hashing via pbkdf2_sha256

  AUTHENTICATION:
    - JWT (JSON Web Tokens) via python-jose

10. WORKFLOW: HOW A USER QUERY IS PROCESSED
================================================================================

  Example: Student asks "When can I meet Shreya mam?"

  1. Frontend sends POST /chat/query?query=When+can+I+meet+Shreya+mam
     with JWT token in Authorization header.

  2. Backend verifies JWT → identifies user as "student" role.

  3. QueryRouter.classify() detects "meet" and "mam" → returns "HYBRID".

  4. HYBRID path triggers:
     a. SQL: search_faculty_by_name("shreya", db) → finds StaffProfile
        where name contains "shreya" → returns availability, schedule.
     b. RAG: rag_engine.search("When can I meet Shreya mam") → returns
        top-5 relevant chunks from the vector database.

  5. System prompt is constructed with:
     - User role: student
     - SQL data: {name: "Shreya", dept: "ISE", available: true,
                  schedule: [{day: "Monday", start: "10:00 AM", end: "11:30 AM"}]}
     - RAG context: relevant campus knowledge chunks

  6. Groq API (Llama-3.1-8b-instant) generates a response:
     "Shreya is currently available. You can meet her on Monday
      from 10:00 AM to 11:30 AM. She is in the ISE department."

  7. Response sent back to frontend with intent="HYBRID" and source tag.


11. ADMIN KNOWLEDGE MANAGEMENT WORKFLOW
================================================================================

  A. SNAP KNOWLEDGE (Quick entry, no category)
  ─────────────────────────────────────────────
  Admin types: "DSCE won Best Engineering College Award 2025"
  → Saved to data/documents/ as text file
  → Chunked and embedded by RAG engine
  → Immediately searchable in the vector database
  → Students can now ask about the award

  B. FORMAL KNOWLEDGE (Structured entry with category)
  ────────────────────────────────────────────────────
  Admin fills:
    Title: "Attendance Rules 2025"
    Category: "Rules"
    Content: "Students must maintain 85% attendance..."
  → Same pipeline as above, but tagged with category

  C. FILE UPLOAD (Bulk document indexing)
  ────────────────────────────────────────
  Admin uploads a PDF or TXT file
  → File saved to data/documents/
  → Text extracted and indexed into vector database
  → All content becomes immediately queryable


12. KEY DESIGN DECISIONS
================================================================================

  1. WHY FAISS OVER CHROMA/PINECONE?
     - No external service needed (runs locally)
     - Faster for small-to-medium datasets (<100K vectors)
     - No API costs or rate limits

  2. WHY GROQ OVER OPENAI?
     - Free tier with generous limits
     - Extremely fast inference (LPU hardware)
     - Llama-3.1-8b-instant is sufficient for Q&A

  3. WHY KEYWORD ROUTER OVER LLM ROUTER?
     - Zero latency (instant classification)
     - No extra API call per query
     - More predictable behavior

  4. WHY REACT OVER STREAMLIT?
     - Professional, production-quality UI
     - Full control over design and UX
     - Persistent login via localStorage
     - Faster, more responsive interface

  5. WHY COSINE SIMILARITY OVER L2 DISTANCE?
     - Better for text embeddings (direction matters more than magnitude)
     - Normalized vectors give consistent similarity scores
     - Industry standard for semantic search
